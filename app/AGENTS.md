# App Router Rules

## Module Context

- 이 폴더는 현재 앱의 주 진입점이다.
- `page.tsx`는 루트 화면을 담당한다.
- `layout.tsx`는 전역 레이아웃과 메타데이터를 담당한다.
- `globals.css`는 전역 스타일 토큰과 기본 스타일을 담당한다.
- 현재 앱 UI는 아직 초기 단계이며, 제품 화면은 `app/page.tsx`를 중심으로 발전시킨다.

## Tech Stack & Constraints

- 프레임워크: Next.js App Router
- 언어: TypeScript, TSX
- 스타일링: Tailwind CSS v4와 `globals.css`
- LLM UI가 추가되더라도 API 키와 서버 호출은 이 폴더의 클라이언트 코드에 직접 넣지 않는다.
- 기본 원칙: 서버 컴포넌트를 우선하고, 브라우저 상태나 이벤트가 필요할 때만 `use client`를 추가한다.

## Implementation Patterns

- 페이지는 기본적으로 하나의 화면 책임만 가진다.
- 레이아웃은 공통 구조, 폰트, 메타데이터처럼 전역 책임만 둔다.
- 화면 로직이 커지면 `app/` 안에 세그먼트별 폴더나 별도 컴포넌트로 분리한다.
- 클래스 문자열이 과도하게 길어지면 반복 블록을 작은 컴포넌트로 나눈다.
- 메타데이터는 실제 제품 목적에 맞게 유지하고, 기본 생성값을 오래 방치하지 않는다.
- 채팅 UI를 만들 때는 타임라인, 입력창, 로딩 상태를 분리 가능한 단위로 설계한다.

## Testing Strategy

- 현재 저장소에는 테스트 스크립트가 없다.
- 이 영역을 수정한 뒤에는 최소 `pnpm lint` 기준으로 확인한다.
- UI 복잡도가 커지면 그 시점에 테스트 도구를 추가하고 규칙을 확장한다.

## Local Golden Rules

- `page.tsx`를 데모 템플릿 상태로 오래 두지 않는다.
- 접근성을 위해 의미 있는 HTML 요소와 적절한 텍스트 라벨을 유지한다.
- 전역 스타일은 꼭 필요한 값만 둔다. 화면 전용 스타일은 가능한 한 해당 컴포넌트 근처에서 해결한다.
- 서버 전용 로직이나 비밀값 처리는 이 폴더의 클라이언트 컴포넌트에 직접 넣지 않는다.
- `app/api/`가 생기기 전까지는 API 규칙을 이 파일에 과도하게 넣지 않는다.
